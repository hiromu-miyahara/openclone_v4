# OpenClone 要件定義書 v1.1（ハッカソン版）

## 1. 文書の目的
本書は、個人向け分身AIサービス「OpenClone」のコンセプト、機能要件、非機能要件、MVP受入基準を定義する。

## 2. サービスコンセプト
ユーザー本人の分身として、スマートフォン上で常時開いて横に置いて使える日常チャット型AIエージェントを提供する。  
価値の中核は「正答率」より「本人らしさ（話し方・思考傾向・声質）」の再現性に置く。

## 3. ターゲットと利用シーン
- ターゲット: 個人ユーザー（本人利用）
- 最優先シーン: 日常チャット
- 利用形態:
  - スマートフォンでPWAを起動し、常時表示で会話する
  - 必要に応じて音声再生付きでやり取りする

## 4. プロダクト方針
- 本人らしさ優先:
  - オンボーディング時のヒアリングと音声収集を重視し、人格再現の土台を作る
- スピード重視:
  - 発話開始レイテンシーは1.0秒以下を目標値とする
- 実装方針:
  - 初期はクラウドAPI中心で実装する
  - 学習データのクラウド保存は許容
  - LLMはMistralのモデル群を採用する
  - 音声再現はElevenLabsを採用する
  - クラウド環境はGCPを採用する
  - OpenClawをバックエンド実行基盤として利用し、OpenCloneはラッパーインターフェースとして提供する
  - モデル選定は以下アナウンス準拠とする:
    - Ministral 3（3B/8B/14B）
    - Mistral Large 3
    - Voxtral / Voxtral Realtime

## 5. 機能要件
### 5.0 OpenClawラッパー方針
- OpenCloneはユーザー向けの専用UI/UX（PWA）とAPIラッパーを提供する
- OpenClawはバックエンド側で実行し、エージェント実行・モデル接続・チャネル制御の基盤として利用する
- OpenClaw実行基盤はユーザーごとに専用 Compute Engine インスタンスを割り当てる
- ユーザー登録時に専用 Compute Engine インスタンスを自動プロビジョニングし、OpenClaw導入と `Mistral Large 3` 利用設定を行う
- 会話リクエストはユーザーIDに紐づく専用 OpenClaw インスタンスに必ずルーティングする
- OpenClone固有機能（人格プロファイル、2等身アバター、Voxtral STT、ElevenLabs TTS）はラッパー層で管理する
- 音声I/O責務は以下とする:
  - ユーザー音声入力はアプリ側でVoxtral STTによりテキスト化してOpenClawへ渡す
  - OpenClawのテキスト出力をElevenLabs TTSで音声化し、テキストと音声を同一ターンでアプリへ返す

### 5.1 オンボーディング機能
- 初回セットアップで以下を収集できること
  - 基本プロフィール（口調、呼称、価値観）
  - 思考傾向（意思決定の軸、優先順位、NG表現）
  - サンプル会話テキスト
  - 音声サンプル（TTS声質調整用）
- ヒアリング質問数は8問（固定）とし、回答は音声入力を必須とする
- 音声8問の完了後に、Big5に関する質問を10問（固定）実施する
- Big5回答から5因子スコアを算出し、事前定義のタイプカテゴリ（例: `Explorer`, `Analyst`, `Supporter`, `Driver`）へ割り当てる
- オンボーディング完了画面で「あなたはxxタイプです」を表示する
- 質問セットは固定質問として定義し、MVP中は動的分岐を行わない
- ElevenLabsの声質再現データ確保のため、オンボーディング音声回答は必須収集とする
- Voice Cloneはオンボーディング音声8問の全音声を連結して作成する（品質フィルタリングは行わない）
- ヒアリング結果を「人格プロファイル」として保存し、生成に反映できること
- 音声8問の回答要約とBig5結果（スコア・タイプ）を、人格再現根拠として OpenClaw の `SOUL.md` に反映する
- 人格プロファイルはオンボーディング完了時点で確定し、MVP中は自動更新しない
- 顔写真の品質ゲートは導入するが、閾値詳細は後続バージョンで定義する


### 5.1a Big5実装詳細（TIPI-J準拠）
- 尺度は TIPI-J 10項目（各次元2項目: 正方向1 + 逆転1）を採用する
- 回答スケールは 7件法（1..7）とする
- 逆転項目は #2, #6, #8, #9, #10 とし、`reversed = 8 - raw` で処理する
- 次元スコア（1.0..7.0）は以下で計算する
  - `E = (Q1 + rev(Q6)) / 2`
  - `A = (rev(Q2) + Q7) / 2`
  - `C = (Q3 + rev(Q8)) / 2`
  - `N = (Q4 + rev(Q9)) / 2`
  - `O = (Q5 + rev(Q10)) / 2`
- 正規化スコア（0.0..1.0）は `normalized = (dimension_score - 1.0) / 6.0` を用いる
- 性格タイプは以下のルールで分類する
  - `Leader`: `E >= 0.6 and C >= 0.6`
  - `Supporter`: `A >= 0.6 and N <= 0.4`
  - `Creator`: `O >= 0.6 and E >= 0.5`
  - `Analyst`: `C >= 0.6 and O >= 0.5`
  - `Communicator`: `E >= 0.6 and A >= 0.5`
  - 上記に当てはまらない場合は `Balanced`（または優勢次元に基づくフォールバック）
- トーンパラメータは以下の線形合成で算出する
  - `Formality = 0.4*C + 0.3*(1-E) + 0.3*(1-O)`
  - `Energy = 0.4*E + 0.3*O + 0.2*(1-N) + 0.1*A`
  - `Directness = 0.4*(1-A) + 0.3*C + 0.2*E + 0.1*(1-N)`
  - `Warmth = 0.5*A + 0.2*E + 0.2*(1-N) + 0.1*O`
- 画面表示は 5因子スコア・タイプ名・トーンパラメータを最低限表示する

### 5.2 テキスト生成機能（人格再現）
- 会話文脈と人格プロファイルを入力し、本人らしい応答文を生成できること
- 応答時に以下を優先できること
  - 語彙・言い回しの一貫性
  - 意思決定スタイルの一貫性
  - 口癖・テンポの再現
- 会話ログは表示・検索・削除の対象とし、MVP中は人格プロファイル自動更新に利用しない

### 5.3 音声再現機能
- ElevenLabs TTSで応答文を音声化できること
- テキストと音声を同一ターンで出力できること

### 5.4 ビジュアル機能
- 顔写真1枚から、本人特徴を残した2等身の全身デフォルメドット絵ベースを生成できること
- モーションごとに、ベース全身画像を参照してフレーム画像を生成できること
- 1モーションあたりのフレーム数はMVPで `6` 枚を基本とすること
- 生成フローは `Nanobanana2(全身ベース生成) -> Nanobanana2(ポーズ差分フレーム生成) -> GIF化` の順で処理すること
- 複数モーションパターン（例: idle / thinking / speaking / nod / agree / surprised）を持つこと
- 主要モーションは事前生成したフレーム群で構成すること（MVPではランタイム再生成しない）
- LLM応答時にテキストとモーションアクションを分離して出力し、状態連動アニメーションを切り替えできること
- 画像生成方針はNanobanana2パイプラインを標準採用し、A方式（MediaPipe + OpenCV + Pillow）は代替実装候補として保持すること

### 5.5 UI/UX機能（PWA）
- PWAとしてホーム画面追加できること
- スマホ縦画面を基準に会話UIが崩れず表示されること
- 片手操作を想定した入力・送信導線を持つこと
- テキスト入力と音声入力を同一画面で切り替え利用できること
- メイン画面の初期入力モードはテキスト入力とする
- バックグラウンド復帰後に直前会話を継続できること（少なくとも24時間）
- アセット生成待機は20秒で先行開始を許可し、未完了時は1分/6ページのストーリー演出をループ表示する
- オンボーディング/メイン画面ともにBGMと効果音を導入する

## 6. 非機能要件
### 6.1 性能
- レイテンシー目標:
  - ユーザー入力確定から、テキストまたは音声の応答開始まで1.0秒以下（目標）
- 体感速度改善:
  - ストリーミング出力、短文先行表示、TTS先行再生を適用する
- 測定条件（MVPで採用）:
  - 計測対象端末: 直近3年以内の一般的スマートフォン（iOS/Androidのミドルレンジ以上）
  - 計測回線: Wi-Fi（下り50Mbps以上）および5G
  - 計測点: 送信操作完了から最初の応答文字表示まで
  - 判定指標: p95で1.0秒以下（平均ではなく分位で管理）

### 6.2 可用性
- 通信品質が低下した場合でも、テキスト応答を優先して継続できること

### 6.3 データ管理
- 学習データの保存先はクラウドを許容する
- データ種別（プロフィール、会話ログ、音声）の保存区分を明示する

### 6.4 セキュリティ
- MVP段階では法務文書整備を必須化しない
- ただし最低限として、ユーザー本人のみが編集/利用できる認証導線を実装する
- セーフティ境界（ハッカソンMVP最小要件）:
  - NGカテゴリを拒否する（違法行為支援、差別/ヘイト、露骨な性的コンテンツ、自傷他害の具体的助長、個人情報の不正取得）
  - プロンプトインジェクション対策として、システム指示とユーザー入力を分離し、危険指示は無効化する
  - 個人情報漏えい抑止として、PIIマスキング/抑制チェックはv1.1以降で導入する
  - 外部アクションはMVPでは無効化し、回答専用エージェントとして動作させる
  - OpenClaw実行時はプロンプトインジェクション対策として、`SOUL.md` と system 指示を最上位固定し、ユーザー入力による上書き要求を拒否する
  - Compute Engine 上のホスト情報（metadata / ファイルシステム / 環境変数 / シェル実行）へのアクセス要求を拒否し、会話生成責務のみに制限する

## 7. MVPスコープ
### 7.1 MVPに含める
- 1ユーザー1人格の作成と更新
- オンボーディングヒアリングフロー
- 人格反映テキスト生成
- ElevenLabs TTS連携
- ドット絵アバター表示
- スマホ向けPWA会話画面

### 7.2 MVPに含めない
- 課金実装（将来サブスク想定だが現時点対象外）
- 複数人格切替
- 厳密な法務パッケージ整備
- 完全ローカル学習/推論

## 8. 受入基準（MVP）
- AC-01:
  - 初回オンボーディング（音声8問 + Big5 10問）を完了し、人格プロファイルが保存される
  - オンボーディング完了時点で Voice Clone が作成完了（`voice_profile_ref` 確定）している
- AC-01a:
  - Big5スコアとタイプ分類が計算され、オンボーディング完了画面に表示される
- AC-02:
  - 人格再現ロジック（口調、価値観、口癖、意思決定軸）に基づく応答制御が有効化されている
  - 「本人らしさ」の客観的評価基準はv1.1以降に設定（MVPでは実装担保を優先）
- AC-02a:
  - 音声回答要約とBig5結果が OpenClaw の `SOUL.md` に反映され、会話応答で参照される
- AC-02b:
  - ユーザー登録後に専用 Compute Engine 上の OpenClaw が自動で割り当てられ、他ユーザーと実行基盤を共有しない
- AC-03:
  - 応答開始までの時間が、主要ケースで1.0秒以下を達成する
- AC-04:
  - PWAとしてスマホホーム画面から起動し、会話を継続できる
- AC-05:
  - テキスト応答と音声再生（ユーザー選択時）が動作する
  - 音声生成はデフォルトOFF、ユーザーが「音声再生」ボタンを押した時のみ生成
- AC-06:
  - 2等身ドット絵アニメーションが応答状態およびLLM出力アクションに連動して表示される

## 9. KPI（初期）
- 本人らしさ評価（ユーザー自己評価 5段階）
- 応答速度満足度（5段階）
- 1日あたり継続会話ターン数
- 起動後5分継続率

## 10. 開発優先順位
1. オンボーディングと人格プロファイル設計
2. テキスト生成の人格反映精度向上
3. レイテンシー1秒以下達成
4. TTS連携とドット絵同期
5. PWA最適化と常時利用UX改善

## 11. 今後の拡張方針
- サブスク課金導入
- ローカル推論ハイブリッド化
- 利用同意/規約の正式整備
- 分身品質の自動評価基盤（本人らしさスコア）

## 12. 曖昧点の整理（要件確定チェックリスト）
以下は、曖昧点の確定状況と保留項目の一覧。

1. 「本人らしさ」の判定方法
- 曖昧な点:
  - なし（確定）
- 決めること:
  - MVPでは主観評価の合格ラインを設定しない
  - 人格再現ロジックの実装担保を優先する

2. オンボーディングの深さ
- 曖昧な点:
  - なし（確定）
- 決めること:
  - 質問構成は「音声8問（固定・必須）+ Big5 10問（固定・必須）」
  - 質問は固定セット
  - 回答方式は、音声パートは音声入力必須、Big5パートは選択式
  - 追加プロファイリングはMVPでは実施しない

3. 音声の入出力モード
- 曖昧な点:
  - なし（確定）
- 決めること:
  - 入力方式はMediaRecorder録音 + Voxtral API（iOS対応）
  - 出力方式はテキスト基本、音声はユーザー選択時のみ
  - 画面上でドット絵アニメーションを同時表示する

3.1 ドット絵モーション制御
- 曖昧な点:
  - なし（確定）
- 決めること:
  - ドット絵は2等身アバターで実装する
  - モーションは事前生成フレーム群を使用する
  - LLMのStructured Outputで `text` と `action` を分離出力する
  - MVPはNanobanana2パイプラインを採用し、A方式は代替実装候補として保持する

4. 1秒以下レイテンシーの測定条件
- 曖昧な点:
  - なし（確定）
- 決めること:
  - 対象回線はWi-Fi（50Mbps以上）/5G
  - 対象端末は直近3年以内ミドルレンジ以上
  - 計測点は送信完了から最初の応答文字表示
  - 合格判定はp95で1.0秒以下

5. PWAの常時利用体験
- 曖昧な点:
  - なし（確定）
- 決めること:
  - 画面常時ONは要件にしない（OS仕様依存のためMVP対象外）
  - 通知は任意（許諾時のみ）で、MVP必須要件にはしない
  - 復帰後24時間は会話コンテキストを継続する

6. 会話ログの保存ポリシー
- 曖昧な点:
  - なし（確定）
- 決めること:
  - 保存期間は無期限
  - ユーザーがサービス画面から手動削除できること
  - 削除粒度はメッセージ単位とする

7. 人格の更新ルール
- 曖昧な点:
  - なし（確定）
- 決めること:
  - MVPでは人格プロファイルの自動更新を行わない
  - 人格はオンボーディング完了時点で確定する
  - 再設定はユーザー明示操作でオンボーディング再実行時のみ

11. ブラウザ対応範囲（MVP）
- 曖昧な点:
  - なし（確定）
- 決めること:
  - MVPの対応対象は `iOS Safari` と `Android Chrome` に限定する

8. セーフティ境界
- 曖昧な点:
  - なし（確定）
- 決めること:
  - NG領域を定義し拒否する
  - 拒否時も本人口調は維持しつつ、安全な代替案を返す
  - 参照プラクティス: OWASP Top 10 for LLM Applications（Prompt Injection / Sensitive Information Disclosure / Excessive Agency など）、NIST AI RMF Generative AI Profile

9. 将来のサブスク設計に向けた最小データ
- 曖昧な点:
  - MVPでは検討保留
- 決めること:
  - v1.2以降で定義

10. 品質検証の運用
- 曖昧な点:
  - MVPでは検討保留
- 決めること:
  - v1.2以降で定義
